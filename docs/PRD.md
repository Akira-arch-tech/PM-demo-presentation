# FUNDAMENTAL 定制品设计网站 PRD（业务展示版）

## 0. 项目定义

- 项目名：FUNDAMENTAL（Fun is a kind of mental）
- 项目目标：复刻并优化“定制商品网站”的核心链路，形成可演示、可验证、可扩展的产品方案
- 演示对象：面试官、业务负责人、设计/研发协作团队

## 1. 背景与机会

### 1.1 背景

定制电商的核心矛盾是：  
用户想要“个性化”，但又担心“做出来不好看、价格不透明、交付不确定”。

### 1.2 市场机会

- C 端：礼物、宠物、应援、纪念品需求稳定增长
- B 端：中小商家需要低门槛定制能力，用于小批量活动营销
- 平台侧：定制链路比普通电商更容易形成高客单和复购

### 1.3 项目价值主张

- 对用户：5 分钟内完成“选品 + 设计 + 报价 + 下单”
- 对业务：提升转化率，降低咨询成本，提高首单成交率
- 对团队：先用最小成本验证需求，再逐步接真实产线能力

## 2. 目标用户与使用场景

### 2.1 目标用户

- C 端用户（核心）：
  - 年轻女性、学生、宠物主、应援用户
  - 对“个性化 + 情绪价值 + 可分享”敏感
- B 端用户（扩展）：
  - 小商家、工作室、社群团购组织者
  - 对“快速出样 + 低起订 + 价格可控”敏感

### 2.2 典型场景

1. 送礼场景：用户上传照片 + 文字，快速生成礼物
2. 应援场景：用户制作立牌/钥匙扣，支持批量购买
3. 商家场景：活动前快速定制小批量周边

## 3. 产品目标与衡量指标（KPI）

### 3.1 北极星指标（North Star Metric）

- 定制下单完成率 = 完成下单人数 / 进入定制页人数

### 3.2 阶段目标（Demo -> Beta）

- 阶段 A（当前 Demo 验证）
  - 主流程完成率 >= 80%（从选品走到下单弹窗）
  - 设计操作触发率 >= 70%（至少修改一次文字/颜色/尺寸）
- 阶段 B（小流量试运营）
  - 下单转化率 >= 8%
  - 订单支付成功率 >= 90%
  - 售前咨询率下降 20%（因预览与报价更清晰）

### 3.3 过程指标（可每日追踪）

- 商品点击率（商品卡点击 / 展示）
- 设计完成率（进入设计区后发生至少 2 次编辑）
- 报价触发率（触发报价接口 / 进入设计区）
- 下单提交率（点击下单 / 进入设计区）
- 异常率（报价失败、下单失败）

## 4. 竞品拆解与差异化策略

### 4.1 参考竞品（7sGood 同类能力）

- 强项：
  - 商品池丰富，折扣明显
  - 定制入口明确，营销氛围强
  - 价格锚点清晰（低价吸引）
- 可优化点（作为我们的机会）：
  - 设计与报价逻辑的解释可更透明
  - 新手引导可以更“低学习成本”
  - 定制后的信心建立（效果、工艺、时效说明）可更强

### 4.2 FUNDAMENTAL 差异点（面试可重点讲）

1. **更易懂**：把复杂流程压成一屏四步，降低认知负担  
2. **更透明**：报价拆解展示（单价、小计、服务费、总计）  
3. **更可控**：每次编辑都能马上看到预览反馈  
4. **更可扩展**：前后端已解耦，后续可替换真实渲染与订单系统

## 5. 产品范围（MVP）

### 5.1 功能模块

1. 商品选择：分类筛选 + 商品卡切换  
2. 定制编辑：文字、颜色、尺寸、素材上传  
3. 实时预览：编辑即反馈  
4. 报价计算：实时返回金额  
5. 下单模拟：生成订单号与预计发货

### 5.2 暂不纳入（本阶段不做）

- 登录注册
- 在线支付
- 真实物流追踪
- 复杂图像智能抠图/排版编辑器

## 6. 用户流程与数据流（谁负责什么）

1. 用户在前端选择商品 -> 前端请求商品接口  
2. 用户修改设计参数 -> 前端请求报价接口  
3. 后端返回金额明细 -> 前端更新价格区  
4. 用户点击下单 -> 前端调用订单接口  
5. 后端生成订单号 -> 前端反馈下单结果

责任边界：
- 前端负责“展示与交互”
- 后端负责“规则计算与结果返回”
- 数据层负责“记录商品、报价、订单快照”

## 7. 非功能要求（业务可理解版）

- 速度：首屏加载快，核心操作反馈 < 1 秒（Demo 环境）
- 稳定：接口异常时有可读错误提示，不让用户“卡死”
- 可演示：不登录即可走完整流程
- 可扩展：API 结构可平滑升级为真实业务系统

## 8. 风险评估与应对策略

### 8.1 业务风险

- 风险：用户担心“实物和预览不一致”
- 应对：增加材质说明、样品图、工艺说明和发货时效说明

### 8.2 技术风险

- 风险：图片处理能力不足导致效果不佳
- 应对：M3 引入图像处理服务和对象存储，先保证上传和预览闭环

### 8.3 项目风险

- 风险：需求膨胀导致排期失控
- 应对：坚持 MVP 边界，按周评审“新增需求是否影响主流程”

## 9. 排期计划（4 周版本）

### Week 1：需求与信息架构

- 输出：PRD v1、竞品拆解、关键流程图
- 验收：业务方确认“做什么/不做什么”

### Week 2：前端交互 Demo

- 输出：可点击原型（选品、设计、预览、报价区）
- 验收：可完整走通演示脚本

### Week 3：后端 API 联调

- 输出：商品/报价/下单接口 + 前后端打通
- 验收：接口调用成功率、演示稳定性达标

### Week 4：优化与展示准备

- 输出：指标埋点定义、风险说明、面试讲述材料
- 验收：可进行 10 分钟业务路演 + 5 分钟问答

## 10. 验收标准（DoD）

1. 主流程可跑通：选品 -> 设计 -> 报价 -> 下单  
2. 报价结果与输入参数一致（数量、尺寸影响价格）  
3. 异常有提示，不出现无反馈状态  
4. 文档齐全：PRD、架构拆解、M2 API 说明可用于评审  
5. 可用于面试展示：有目标、有策略、有风险、有排期

## 11. 面试讲述建议（1 分钟版本）

“我做了一个定制电商的 MVP。先拆竞品能力，再把流程收敛成四步：选品、设计、报价、下单。  
我把前后端职责拆清楚：前端负责体验，后端负责规则。  
在结果上，我不仅做了可点击 demo，还补了可落地的指标体系、风险应对和 4 周排期，所以这不是一个静态原型，而是可进入试运营的产品方案。”
