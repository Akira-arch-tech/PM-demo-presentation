# 7sGood（King_13）技术架构拆解（面向产品经理版）

## 1. 我们看到的网站能力

基于页面公开信息与可见交互，`https://www.7sgood.com/category/King_13` 体现了这些能力：

1. 分类浏览：按“亚克力/手机壳/DIY”等品类筛商品。
2. 商品卡展示：名称、价格、折扣、积分等营销信息。
3. 定制商品导流：可进入“上传素材 + 自定义”的流程。
4. 促销体系：折扣、券、免邮阈值等增长机制。
5. App 承接：强引导下载 App（典型增长策略）。

## 2. 用“谁负责什么”理解技术架构

> 下面是产品视角下的“最小全栈分层”，用于做可运行 demo。

### A) 前端层（你看到的网页）

- 负责：页面展示、分类筛选、按钮点击、输入框交互、实时预览。
- 典型技术：`React/Vue`（组件化 UI）、`CDN`（静态资源加速）。
- 对 demo 的关系：你做的交互原型，90% 先在这一层实现。

### B) 网关层（统一入口）

- 负责：接收所有客户端请求，做鉴权、限流、路由到具体服务。
- 公开线索：目标站请求返回头出现 `APISIX`（常见 API Gateway）。
- 对 demo 的关系：前期可不实现，先在文档里标明“未来接入”。

### C) 业务服务层（核心能力）

- 商品服务：类目、SKU、价格、库存、上下架状态。
- 定制服务：模板、材质、尺寸、印刷参数、价格计算规则。
- 订单服务：购物车、下单、支付状态、履约状态。
- 营销服务：优惠券、折扣、活动策略、首单权益。
- 对 demo 的关系：先用“本地假数据 + 简化规则”，后续再替换 API。

### D) 媒资与设计处理层（定制网站关键）

- 负责：上传图片、存储、裁剪、生成预览图、输出生产文件。
- 典型技术名：
  - `Object Storage`（对象存储：专门放图片/文件）
  - `Image Processing`（图像处理：裁剪/压缩/合成）
- 对 demo 的关系：先做“上传占位 + 预览模拟”，不做真实渲染引擎。

### E) 数据层（系统记忆）

- 负责：保存商品、用户、订单、优惠、设计稿记录。
- 典型结构：
  - `MySQL/PostgreSQL`（业务结构化数据）
  - `Redis`（高频缓存）
- 对 demo 的关系：第一版可不接数据库，用 JSON 假数据即可演示流程。

## 3. 关键数据流（最重要）

1. 用户选择商品 -> 前端请求商品服务拿到价格和规格。
2. 用户输入文字/上传图 -> 定制服务生成预览参数。
3. 用户改数量/尺寸 -> 定价服务实时计算金额。
4. 用户点击下单 -> 订单服务创建订单并锁定价格。
5. 订单推给工厂/履约系统 -> 回传物流与状态给前端。

## 4. 复刻时“最小可跑”范围（MVP）

为了快速做演示，建议只做这 4 件事：

1. 商品列表 + 分类筛选
2. 设计面板（文字、颜色、尺寸、上传）
3. 预览面板（实时同步）
4. 报价与下单按钮（流程模拟）

这样你就能讲清楚：
- 用户怎么走流程；
- 系统在哪些节点算价与存数据；
- 后续怎么把“假数据”换成“真实后端”。

## 5. 风险与假设

- 风险：目标站存在反爬策略（直接请求可能 403），公开代码线索有限。
- 假设：按行业通用架构 + 可见能力做拆解，足够支持 PM 级方案设计和 demo 搭建。
- 结论：本方案适用于“功能复刻与流程验证”，不等于源码级逆向。
